# Безопасность и конфиденциальность (NDA)

Данные MacroData и любые данные, передаваемые в ассистент, считаются **конфиденциальными и под NDA**. Запрещено любое извлечение, выгрузка или утечка данных. Ниже — как проект обеспечивает максимальную конфиденциальность.

---

## 1. Запрет извлечения данных (приоритет)

- **Нельзя добыть данные через ассистента.** В системный промпт заложены жёсткие правила: при любом запросе, направленном на получение сырых данных, списков записей, выгрузки, экспорта или копирования — ассистент обязан отказать без исключений.
- **Запрещено в ответах:** полные таблицы, все записи, дампы, экспорт в CSV/JSON/Excel, перечисление записей построчно или пачками, повторение больших объёмов конфиденциальных данных. Разрешены только: структура (таблицы, поля), сводки и агрегаты, рекомендации по метрикам, примеры формата с демо-данными.
- **Серверная проверка:** перед отправкой в модель запрос проверяется на явные формулировки выгрузки/экспорта (см. `app/lib/requestGuard.ts`). При совпадении запрос блокируется (HTTP 403), тело запроса не передаётся в API и не логируется.
- При малейшем сомнении ассистент должен отказать, чтобы исключить утечку или извлечение данных.

---

## 2. Режим «только чтение» (база данных)

- **Никаких изменений в данных.** Приложение предназначено только для **чтения** данных. Запросы к БД должны выполняться **исключительно в режиме SELECT** (или аналог только чтения).
- **Рекомендация при подключении БД:** завести отдельного пользователя БД с правами **только чтение** (например, только `SELECT` по нужным таблицам). Не использовать учётную запись с правами `INSERT`, `UPDATE`, `DELETE`, `ALTER`, `DROP`, `TRUNCATE`.
- В коде ассистента в системный промпт заложено правило: **никогда не предлагать и не выполнять** действия, изменяющие базу или данные. На запросы «сохранить», «удалить», «обновить» данные в БД ассистент отвечает отказом и пояснением про режим только чтения.

---

## 3. Конфиденциальность и отсутствие утечек

- **Сервер не сохраняет и не логирует содержимое запросов и ответов.** В API чата контент сообщений не пишется в логи и не сохраняется на сервере.
- **Ответы не кэшируются.** Для API чата выставлены заголовки `Cache-Control: no-store, no-cache`, `Pragma: no-cache`, чтобы ответы с данными не попадали в кэш прокси/браузера.
- **Заголовки безопасности:** в ответах API установлены `X-Content-Type-Options: nosniff` и отключено кэширование.

---

## 4. Хранение на клиенте (localStorage)

- История чата сохраняется **только в браузере пользователя** (localStorage), не отправляется на сторонние серверы и не передаётся никому, кроме текущего сеанса.
- Для максимальной конфиденциальности пользователь может отключить сохранение истории (при необходимости это можно вынести в настройки) или очищать данные браузера после работы.

---

## 5. Передача данных в OpenAI (LLM)

- Текст запроса и ответ модели передаются в API OpenAI для формирования ответа. По политике OpenAI данные API по умолчанию могут использоваться для улучшения моделей, если не отключено отдельно.
- **Рекомендации для данных под NDA:**
  - В [настройках OpenAI](https://platform.openai.com/settings/organization/data-usage) отключить использование данных API для улучшения моделей (Data usage / Training).
  - Либо использовать **Azure OpenAI** с контрактом, где явно прописано, что данные не используются для обучения и хранятся в нужной юрисдикции.

---

## 6. Рекомендации по развёртыванию

- **HTTPS:** работать только по HTTPS, чтобы трафик (включая запросы и ответы) был зашифрован.
- **Секреты:** все ключи (OPENAI_API_KEY, при подключении БД — учётные данные) хранить в переменных окружения (например, `.env`), не коммитить их в репозиторий. Файл `.env`/`.env.local` должен быть в `.gitignore`.
- **Доступ к серверу:** ограничить доступ к серверу и БД по IP и по ролям; использовать отдельного пользователя БД с минимальными правами (только чтение).
- **Аудит:** при необходимости вести логи только метаданных (например, факт вызова API без тела запроса/ответа) для аудита доступа без риска утечки контента.

---

## 7. Краткий чек-лист

| Требование | Реализация |
|------------|------------|
| Нельзя извлечь/выгрузить данные | Строгий запрет в промпте; серверная проверка (requestGuard) на явные запросы выгрузки; отказ при сомнении |
| Данные не изменяются | Режим только чтение; в промпте — запрет на предложения изменений в БД |
| Нет утечек контента | Контент запросов/ответов не логируется и не сохраняется на сервере |
| Нет кэширования ответов с данными | Заголовки no-store / no-cache в API чата |
| БД только чтение | Подключать БД под пользователем с правами только SELECT |
| Конфиденциальность NDA | Использовать OpenAI с отключённым обучением на данных или Azure OpenAI |

---

## 8. Аутентификация при старте

- Если в окружении заданы **MACRODATA_AUTH_LOGIN**, **MACRODATA_AUTH_PASSWORD** и **MACRODATA_SESSION_SECRET** (не короче 32 символов), доступ к приложению возможен только после входа. Без валидной сессии запросы к `/api/chat` возвращают **401**.
- **Логин и пароль** хранятся только на сервере (переменные окружения), не передаются на клиент и не логируются. Сравнение учётных данных выполняется через HMAC (фиксированное время, без утечки по длине).
- **Сессия:** после успешного входа устанавливается HTTP-only cookie с подписанным токеном (HMAC-SHA256), срок действия 24 часа. Cookie не доступна из JavaScript (защита от XSS), при необходимости задаётся `Secure` и `SameSite=Strict`.
- Рекомендуется задать **сложные логин и пароль** (длина, буквы, цифры, символы) и хранить их в надёжном месте. Пример сгенерированных значений — в `.env.example`; при желании замените на свои и смените **MACRODATA_SESSION_SECRET** (например, `openssl rand -base64 32`).

При появлении новых требований по NDA или безопасности их стоит отразить в этом документе и в коде (промпт, настройки API, доступ к БД).
