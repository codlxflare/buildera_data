### Запрос 1: «должны до 28 февраля… с именами и телефонами»

| Этап | Что в логе | Верно? |
|------|------------|--------|
| REQUEST | Вопрос, stream, historyLength | ✓ |
| AI_SQL_STEP | SQL с `contacts c ON f.contacts_id = c.contacts_id` | ✗ Старый вариант: контакт по `f.contacts_id` — в БД по нему часто пусто |
| SQL_RUN | ok, 46 строк | ✓ |

**Итог по запросу 1:** Запрос выполняется, даты и суммы верные, но имена и телефоны не подтягиваются, потому что в SQL контакт берётся по `finances.contacts_id`. Нужно брать контакт из сделки: `estate_deals.contacts_buy_id → contacts`.

### Запрос 2: «общая сумма задолженности»

| Этап | Что в логе | Верно? |
|------|------------|--------|
| REQUEST | «скажи мне общую сумму задолженности» | ✓ |
| AI_SQL_STEP | **extractedSql: (нет блока sql)** — модель ответила только текстом | ✗ Число взято из контекста чата, не из БД |
| NO_SQL | Ответ: «52,461,231.00 тенге» без выполнения запроса | ✗ Сумма может быть неточной (округление, обрезка таблицы в контексте) |

**Итог по запросу 2:** На вопрос про общую сумму должен выполняться SQL с `SELECT SUM(...)`, а не только текстовый ответ из истории. Иначе цифра не гарантированно совпадает с БД.

---

## Что изменено в коде

1. **Имена и телефоны для finances**  
   В инструкциях явно задано: для списка «с именами и телефонами» по платежам (finances) контакт брать **только из сделки**:
   - `LEFT JOIN estate_deals ed ON f.deal_id = ed.deal_id`
   - `LEFT JOIN contacts c ON ed.contacts_buy_id = c.contacts_id`  
   Вариант с `f.contacts_id` для имён/телефонов использовать нельзя (в БД по нему часто пусто).

2. **«Общая сумма»**  
   На вопросы «общая сумма задолженности», «итого», «суммарно» модель обязана формировать и выводить блок `sql` с запросом к БД (например `SELECT SUM(f.summa) FROM finances f WHERE ...`), а не только писать число из контекста.
