# Чат с данными: точность (text-to-SQL) и экономия токенов

## Text-to-SQL: как достигается точность

Запросы по данным обрабатываются **одним шагом генерации SQL** по тексту пользователя (text-to-SQL):

1. **Схема БД** — в системный промпт подставляется список таблиц и полей (из `api.txt`). Опционально можно использовать **компактную схему** (только ключевые таблицы: сделки, заявки, финансы, дома, контакты и т.д.) через `USE_COMPACT_SCHEMA=true` — меньше токенов, для типичных запросов часто достаточно.

2. **Руководство для ИИ** — `docs/AI_SCHEMA_GUIDE.md`: маппинг «вопрос → таблицы», связи (JOIN), даты и периоды. Это снижает ошибки и угадывание таблиц.

3. **Примеры из БД** — при наличии подключения подставляются примеры значений (`status_name`, `channel_name`, `deal_date` и т.д.) из `getSchemaSamplesText()`, чтобы ИИ использовал реальные значения в условиях.

4. **Повтор при ошибке** — если SQL упал с ошибкой или вернул 0 строк (а запрос подразумевал данные), выполняется повторный вызов с контекстом ошибки и при необходимости подсказкой по таблице `finances` и JOIN.

5. **Валидация** — только `SELECT`, только разрешённые таблицы из белого списка, запрет опасных конструкций (`sqlRunner.ts`).

Итог: один мощный вызов (по умолчанию `gpt-4o`) генерирует SQL; второй вызов — уже **без схемы БД** — только форматирует результат (таблицы, вывод, графики). Так мы не дублируем тяжёлый контекст на этапе форматирования.

## Экономия токенов

### Разные модели под задачи

| Задача | Переменная окружения | По умолчанию | Зачем |
|--------|----------------------|--------------|--------|
| Генерация SQL (text-to-SQL) | `OPENAI_CHAT_MODEL` или `OPENAI_SQL_MODEL` | `gpt-4o` | Точность и сложный контекст (схема, руководство). |
| Форматирование ответа | `OPENAI_FORMAT_MODEL` | `gpt-4o-mini` | Меньше контекста (только данные + инструкции), можно дешевле. |
| Суммаризация истории диалога | `OPENAI_SUMMARY_MODEL` | `gpt-4o-mini` | Один короткий запрос для сжатия старых сообщений. |

Рекомендация: оставить SQL на `gpt-4o` (или аналог), формат и суммаризацию — на `gpt-4o-mini`.

### Компактная схема для SQL

- `USE_COMPACT_SCHEMA=true` — в промпт для генерации SQL подставляется только набор ключевых таблиц (`getMacrodataSchemaShort()`), а не полный список из `api.txt`.
- Экономит сотни/тысячи токенов на каждый запрос с данными.
- Если запросу нужна редкая таблица — можно отключить и использовать полную схему, либо доработать список таблиц в `apiSchema.ts` (SHORT_TABLES).

### Экономичная память диалога

- **Обычный режим:** в запрос передаются последние N сообщений (до ~10k токенов); при переполнении старые пары user/assistant отбрасываются.
- **С суммаризацией** (`ENABLE_DIALOG_SUMMARY=true`): при длинной истории (например, >12 сообщений) старые реплики один раз сжимаются дешёвой моделью в краткий текст «Контекст диалога: …». В запрос уходят этот контекст + последние 6 пар. Так диалог «помнится» целиком при меньшем числе токенов.
- Суммаризация добавляет один дополнительный вызов к API при каждом новом сообщении, если история длинная; включать по необходимости.

### Пример .env.local

```bash
# Обязательно
OPENAI_API_KEY=sk-...

# Точность SQL (по умолчанию gpt-4o)
OPENAI_CHAT_MODEL=gpt-4o

# Экономия: форматирование и суммаризация на дешёвой модели
OPENAI_FORMAT_MODEL=gpt-4o-mini
OPENAI_SUMMARY_MODEL=gpt-4o-mini

# Меньше токенов на шаг SQL — только ключевые таблицы
USE_COMPACT_SCHEMA=true

# Сжимать длинную историю в контекст (один доп. запрос при длинном чате)
ENABLE_DIALOG_SUMMARY=false
```

## Кратко

- **Точность:** полная/компактная схема + AI_SCHEMA_GUIDE + примеры из БД + повтор при ошибке/пустом результате.
- **Экономия:** отдельная модель для форматирования и (опционально) для суммаризации; компактная схема; сжатие длинной истории в один контекст.
